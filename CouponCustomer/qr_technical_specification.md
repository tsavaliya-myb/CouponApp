# QR System Technical Specification

This document explains the technical implementation of the QR code identity system used in the CouponApp for user verification and coupon redemption.

## 1. QR Payload Structure

The QR code contains an **AES-256 encrypted JSON string**. Decrypting this string reveals the following fields:

| Field | Description | Source |
| :--- | :--- | :--- |
| [uid](file:///e:/CouponApp/CouponCustomer/lib/features/qr/presentation/screens/qr_screen.dart#217-265) | **User ID** - The unique identifier of the customer. | `UserModel.id` |
| `tok` | **Subscription/Auth Token** - A token verifying the user's active session. | App Secure Storage (`access_token`) |
| `iat` | **Issued At** - Timestamp (ms) when the QR was generated. | `DateTime.now()` |
| `type` | **Payload Type** - Fixed value to identify the QR purpose. | Constant: `"USER_PROFILE"` |

## 2. Security Layers

### AES-256 Encryption
To prevent users from spoofing IDs by editing QR screenshots, the payload is encrypted using a **Secret Key** shared between the Customer App, Seller App, and Backend.
*   **Cipher**: AES-256-CBC
*   **Encoding**: Base64 (to make the encrypted bytes representable as text in a QR)

### Short-lived TTL (Time-To-Live)
The `iat` (Issued At) field is used to enforce a **5-minute validity window**. Even if a user takes a screenshot of their QR and shares it, that screenshot will become invalid after 5 minutes because the `iat` timestamp will be too old.

## 3. Step-by-Step Flow

### Phase A: QR Generation (Customer App)
1.  **State Check**: The app verifies the user has an `ACTIVE` subscription status.
2.  **Payload Assembly**: The [QrTokenService](file:///e:/CouponApp/CouponCustomer/lib/core/security/qr_token_service.dart#9-48) gathers the [uid](file:///e:/CouponApp/CouponCustomer/lib/features/qr/presentation/screens/qr_screen.dart#217-265), `accessToken`, and current `iat`.
3.  **Encryption**: The JSON is encrypted with AES-256 and converted to a Base64 string.
4.  **Display**: The [QrScreen](file:///e:/CouponApp/CouponCustomer/lib/features/qr/presentation/screens/qr_screen.dart#7-13) uses the `qr_flutter` package to render the Base64 string as a QR code.
5.  **Auto-Refresh**: Every 5 minutes, the app automatically re-generates the QR with a fresh `iat` timestamp, causing the QR pattern to change.

### Phase B: Scanning & Verification (Seller App)
1.  **Capture**: The seller scans the QR using the mobile camera.
2.  **Decryption**: The seller app uses the shared **Secret Key** to decrypt the Base64 string back into JSON.
3.  **Local Validation**:
    *   **Expired?**: If [(CurrentTime - iat) > 5 minutes](file:///e:/CouponApp/CouponCustomer/lib/features/qr/presentation/screens/qr_screen.dart#17-214), the app shows an "Expired QR" error.
    *   **Valid Type?**: If `type` is not `"USER_PROFILE"`, it rejects the scan.
4.  **API Verification**: The seller app calls the backend API (e.g., `/api/redemption/verify-user`) with the [uid](file:///e:/CouponApp/CouponCustomer/lib/features/qr/presentation/screens/qr_screen.dart#217-265) and `tok` found in the QR.
5.  **Final Confirmation**: The backend confirms:
    *   The user is real.
    *   The user has an active coupon book for the seller's city.
    *   The user is not blocked.
6.  **Seller UI**: The seller app displays the **User's Name** and **Profile Photo** to the merchant for visual confirmation before proceeding to coupon selection.

## 4. Why the "Subscription Token" is Important
The `tok` (Subscription Token) acts as a secondary layer of proof. Even if someone managed to decrypt the QR, they cannot simulate a valid subscription without a token signed by the backend. This ensures that the QR can only be generated by a truly authenticated and authorized device.
